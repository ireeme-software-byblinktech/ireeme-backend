PART 1 — WHAT YOU ARE
BUILDING
Overview • System Architecture • User Portals
What is Blink Campus?
Blink Campus is a school management platform. You are building the BACKEND ONLY — the API server
that all 9 types of users (portals) connect to. Each school is fully isolated: a student from
School A can never see data from School B. This isolation is enforced automatically on every query.
The 9 User Portals
Portal User Type Main Capabilities
Student Enrolled students View grades, submit assignments, check attendance, library, health
Teacher Class teachers Create assignments, mark grades, record attendance, manage notes
School Admin School administrator Manage students/teachers, timetable, elections, documents
Super Admin Platform owner Manage ALL schools and school admins across the entire platform
Nurse School health officer Health records, appointments, medical cases, medications
Discipline Discipline officer Cases, scoring, appeals, offense rules, permissions
Accountant Finance staff Fee records, transactions, stock management, staff salary
Parent Parents / guardians Monitor child's grades, attendance, health, pay fees

PART 6 — RULES TO FOLLOW
Architecture • Security • Testing • Naming Conventions
Architecture Rules
Always filter by school_id
Every Prisma query must include where: { schoolId }. Forgetting this
allows cross-school data leaks.
Never skip the layer chain
Controller → Service → Repository. No DB queries in controllers. No
business logic in repositories.
Modules talk via Services only
If Grades needs student data, import StudentService. Never import
StudentRepository directly.
All list endpoints must paginate
Never return an unbounded list. Use cursor or offset pagination.
Limit: 25-50 items max per page.
DTOs on all input
Define DTO class with class-validator decorators. Global
ValidationPipe rejects bad input with 400.
Short transactions
Never call external APIs inside a DB transaction. Keep transactions
fast and atomic.
Events for side effects
Grade posted → emit GradePosted event → NotificationService
creates notification. Never call notification directly from grades
service.
Security Rules
Never store secrets in code
API keys, JWT secrets, DB passwords → .env file only. Never
commit .env to Git.
Passwords → bcrypt always
bcrypt.hash(password, 12) on save. bcrypt.compare() on login.
Never plain text.
File uploads: validate twice
Check MIME type AND file magic bytes. Extension alone is not
enough. Max 10MB.
Return 404 not 403 for wrong school
Don't reveal that another school's data exists. Always 404 if not in
your school.
Refresh tokens → hashed in DB
Store bcrypt hash of refresh token, not the token itself. Compare on
refresh.
Blacklist logout tokens
On logout: store the access token JTI in Redis with TTL = remaining
token lifetime.

Attack Prevention Summary
SQL Injection
Structurally impossible — Prisma uses parameterized queries. Raw
queries use tagged template literals.
XSS
API is JSON-only. No HTML rendered server-side. All file content stored
in S3 with Content-Disposition: attachment.
CSRF
Access tokens are Bearer (not cookies). Refresh cookie uses
SameSite=Strict + custom header check.
Brute Force
10 failed logins → 15-min Redis lockout. 20 failures in 1hr →
admin-unlock required.
Path Traversal
All uploaded files renamed to UUIDv4. Original filename never used in
storage path.
Unauthorized Access
school_id on every query. Wrong school = 404. Wrong role = 403. No
token = 401.
